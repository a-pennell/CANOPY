# Canopy Unified Shell Information Architecture

## Purpose

The Canopy shell is the shared civic and ecological operating environment that makes Canopy feel like one system rather than a collection of inherited products.

It is not a landing page, dashboard wrapper, or navigation bar over separate modules. It is the user-facing expression of the Canopy kernel: identity, scope, object graph, claims and evidence, governance, civic memory, attention, data stewardship, and federation.

The shell must make these facts obvious in use:

- Everything important is an object in a shared civic/ecological graph.
- Every object belongs to one or more scopes: organization, place, commons, living system, project, mandate, or federation context.
- Every consequential action has authority, evidence, memory, and an appeal or review path.
- Old project boundaries disappear into Canopy capabilities.
- The cybernetic loop is visible everywhere: observe, understand, simulate, deliberate, coordinate, act, learn.

## Core IA Thesis

Canopy's primary interface is not a set of apps. It is a scoped object environment.

Users should not ask, "Which app do I open for this?" They should ask:

- What object am I looking at?
- What scope am I acting within?
- What is happening that needs attention?
- What is known, contested, decided, promised, or learned?
- What capability surface do I need for the next action?

The shell therefore organizes Canopy around:

- A global scope switcher
- Global object search
- Attention inbox
- Object pages
- Civic memory
- Capability surfaces
- Command palette
- Map / graph / list triad
- Decision packets
- Data stewardship and federation controls

## Shell Principles

### 1. One Environment, Many Capabilities

Capabilities are not separate products. Reality Map, Commons Registry, Living Systems, Claims and Evidence, Deliberation, Agreements and Policies, Allocation and Accounting, Flows, Simulation, Civic Memory, Learning, Care Coordination, and Federation are surfaces inside one environment.

The shell may expose capability entry points, but these entry points must never behave like disconnected applications with separate objects, separate histories, separate search, separate permissions, or separate language.

### 2. Scope Before Feature

The first question in Canopy is not "what tool do you want?" It is "where and under whose authority are you acting?"

Scope determines:

- Visible objects
- Local terminology
- Governance rules
- Data stewardship rules
- Available actions
- Relevant indicators and thresholds
- Federation context
- Civic memory filters

### 3. Objects Before Screens

Every persistent civic, ecological, governance, or coordination entity should resolve to an object page. Screens that do not anchor to objects should be temporary workspaces, inbox views, search results, command flows, or capability-specific composition surfaces.

### 4. Memory Before Feed

Canopy should not feel like a social feed. Activity is shown as civic memory: structured, filterable, accountable events tied to objects, claims, decisions, authority, data rules, and outcomes.

### 5. Stewardship Before Ownership

The shell must consistently distinguish stewardship, guardianship, role authority, legal ownership, use rights, obligations, and mandates. Ownership may appear where relevant, but it must not become the default relation.

### 6. Contestability Everywhere

Search results, object facts, AI summaries, model outputs, indicators, decisions, and recommended actions should show how they can be challenged, reviewed, corrected, appealed, or superseded.

### 7. Local Language, Shared Grammar

The shell may show local terms, but canonical object types and relationships remain underneath. A local "water council issue," "maintenance ticket," or "mutual credit offer" must still map to Canopy objects and civic memory events.

## Top-Level Shell Anatomy

The desktop shell has five persistent regions:

1. Global header
2. Scope rail
3. Primary workspace
4. Context panel
5. Attention and command layer

On smaller screens these collapse into a bottom navigation bar, a compact scope switcher, and modal context layers, but the IA remains the same.

### 1. Global Header

Purpose:

Provide stable access to identity, scope, search, commands, attention, and memory without privileging any inherited module.

Required elements:

- Canopy instance identity
- Active scope switcher
- Global object search
- Command palette trigger
- Attention inbox trigger
- Civic memory trigger
- User/session menu
- Data visibility indicator when viewing sensitive scopes

Rules:

- The header does not contain product tabs named CommonCredit, ICOS, Sensemaking, or Stewardship.
- The header search is always object search, not page search.
- The active scope must be visible before users take consequential actions.

### 2. Scope Rail

Purpose:

Let users move among the contexts in which they have membership, authority, stewardship, guardianship, or observation rights.

Scope types:

- Personal work context
- Organization
- Place
- Commons
- Living system
- Project or routine
- Governance process
- Federation partner or shared compact

Rail behavior:

- Shows pinned scopes and recent scopes.
- Distinguishes membership from stewardship, guardianship, mandate, and observer access.
- Shows unresolved attention counts per scope.
- Allows quick switching without losing the current object when that object is visible in the new scope.
- Provides a "scope map" view for nested relationships such as neighborhood, city, watershed, bioregion, commons, and federation.

Rules:

- Scope is not a folder. One object can appear in many scopes.
- Switching scope changes permissions and relevance, not the object's identity.
- Administrative boundaries and ecological boundaries must coexist. Watersheds, foodsheds, habitats, and bioregions cannot be hidden behind municipal hierarchy.

### 3. Primary Workspace

Purpose:

Render the current object, capability surface, search result set, map, graph, list, decision packet, or civic memory view.

Workspace modes:

- Object page
- Capability surface
- Search results
- Map view
- Graph view
- List view
- Decision packet
- Civic memory
- Composition flow
- Review workspace

Rules:

- Object pages are the default destination for persistent objects.
- Capability surfaces should open around objects and scopes, not around old product ancestry.
- Composition flows must write draft objects, claims, proposals, commitments, or events rather than isolated form submissions.

### 4. Context Panel

Purpose:

Show the relationships, authority, evidence, risks, thresholds, and next actions that explain the current workspace.

Typical panels:

- Object summary
- Scope and relationship path
- Claims and evidence
- Authority and mandates
- Relevant policies
- Indicators and thresholds
- Active issues and proposals
- Commitments and obligations
- Data stewardship
- Related civic memory
- Suggested next actions

Rules:

- The context panel should make consequences legible before action.
- AI summaries may appear here, but must expose inputs, uncertainty, and challenge paths.
- The panel should adapt to object type without breaking the universal object grammar.

### 5. Attention And Command Layer

Purpose:

Provide two cross-system overlays: one for what needs attention, one for what can be done.

The attention inbox answers:

- What needs review, response, verification, decision, care, or action?

The command palette answers:

- What can I do from here, under this scope and authority?

Rules:

- Attention items and commands are permission-aware.
- Commands should create or mutate canonical objects through governed workflows.
- The command palette should never expose internal service names or old project names.

## Navigation Model

Canopy navigation has three tiers.

### Tier 1: Global Operating Modes

These are stable entry points in the shell:

- Home
- Search
- Attention
- Map
- Graph
- Lists
- Civic Memory
- Capabilities

Home is not a marketing dashboard. It is the user's scoped operating console: current scopes, urgent attention, recently touched objects, active commitments, open governance processes, threshold alerts, and learning reviews.

### Tier 2: Capability Surfaces

Capabilities are domain work surfaces over the shared object graph:

- Reality Map
- Commons Registry
- Living Systems
- Needs and Capabilities
- Claims and Evidence
- Deliberation
- Agreements and Policies
- Allocation and Accounting
- Flows
- Simulation and Models
- Care Coordination
- Learning and Accountability
- Federation

Rules:

- Capability surfaces are verbs and lenses, not product silos.
- Every capability must read and write canonical object references.
- Every consequential capability action must write to civic memory.
- Every capability must participate in at least one phase of the cybernetic loop.

### Tier 3: Object-Local Navigation

Inside an object page, navigation follows the universal object anatomy:

- Overview
- Current state
- Relationships
- Claims and evidence
- Governance and authority
- Commitments and obligations
- Ecological context
- Activity and civic memory
- Outcomes and learning
- Data stewardship and permissions

Object-type variants can emphasize relevant sections, but cannot remove the underlying structure when the section is applicable.

## Scope Switcher

The scope switcher is one of the most important shell controls. It prevents Canopy from collapsing into generic global software by keeping place, authority, ecology, and membership in view.

### Scope Switcher Requirements

The switcher must show:

- Current scope name
- Scope type
- Parent and child scopes
- User relation to the scope
- Active mandate or role, if any
- Data visibility state
- Federation state, if shared across instances
- Local terminology profile

### Scope Selection Model

Users can select:

- Single scope: "Eastside Food Commons"
- Nested scope: "Eastside Food Commons within Cleveland Foodshed"
- Comparative scope: two or more peer places, commons, or projects
- Federation scope: objects shared by multiple instances
- Personal scope: commitments, attention, saved views, drafts

### Scope Effects

Changing scope affects:

- Search ranking and filtering
- Object visibility
- Command availability
- Default map extent
- Civic memory filters
- Local terms
- Permission explanations
- Attention priority
- Applicable policies
- Governance process defaults

### Scope Anti-Patterns

Do not:

- Treat scope as a simple workspace folder.
- Hide ecological scopes under administrative scopes.
- Let users act without showing the authority context.
- Duplicate objects per scope when relationship links are sufficient.
- Use old project names as scopes unless they are historical source labels in migration/admin views.

## Global Object Search

Search is the fastest path through the shell. It must search Canopy's shared object graph, not module-specific records.

### Searchable Objects

Search includes:

- Actors: Person, Account, Organization, Membership, Role, Mandate, Delegation, Guardian
- Reality: Place, Commons, LivingSystem, Resource, Stock, Flow
- Epistemics: Claim, Counterclaim, Evidence, Source, Perspective, Model, Scenario
- Governance: Issue, Proposal, Decision, Agreement, Policy, Appeal, Conflict
- Coordination: Need, Capability, Request, Offer, Commitment, Allocation, Obligation, UseRight
- Action: Project, Routine, Task, Contribution, MaintenanceCycle, RestorationCycle
- Learning: Outcome, Indicator, Threshold, Audit, Retrospective, FeedbackLoop

### Search Result Anatomy

Each result should show:

- Object name
- Canonical type and local term
- Scope chips
- Current state
- Steward, guardian, or authority relation where applicable
- Claim/evidence confidence or contestability signal
- Active attention state
- Visibility and access status
- Last significant civic memory event

### Search Modes

Search supports:

- Direct object lookup
- Semantic search across evidence, deliberations, memory, and policies
- Relationship search: "resources affected by this policy"
- Attention search: "claims needing review"
- Authority search: "objects I can approve"
- Geographic search
- Time-bounded memory search
- Federation search across shared scopes

### Search Rules

- Search must distinguish facts, claims, evidence, model outputs, decisions, and AI summaries.
- Restricted results may appear as redacted stubs when data stewardship rules allow awareness without disclosure.
- Search should rank active scope first, then related scopes, then federated scopes.
- Search should always provide a route to object pages, not isolated result detail screens.

## Object Pages

Object pages are the primary unifying interface pattern. A Resource, Claim, LivingSystem, Policy, Commitment, Model, Flow, Issue, and Decision should feel like different object types in one world.

### Universal Object Page Header

The header must include:

- Object name/title
- Canonical object type
- Local term or alias
- Scope chips
- Steward/guardian/authority relation
- Status
- Last verified or updated date
- Visibility state
- Primary actions

Header rule:

The page must not collapse stewardship into ownership. If legal ownership is relevant, it appears as one relationship among others.

### Universal Object Page Sections

Every object page follows this structure:

1. Current state
2. Relationships
3. Claims and evidence
4. Governance and authority
5. Commitments and obligations
6. Ecological context
7. Activity and civic memory
8. Outcomes and learning
9. Data stewardship and permissions

### Object-Local Actions

Every object page exposes three action zones:

- Observe: add observation, upload source, record condition, record indicator, record flow
- Govern: open issue, create proposal, add perspective, request review, file appeal
- Coordinate: create request, make offer, create commitment, assign task, log contribution, create allocation

Actions appear only when permission checks allow them. Denied actions should explain the authority source and appeal path where safe.

### Object Page Variants

Object-specific emphasis:

- Resource: condition, use rights, maintenance, policies, ecological context
- LivingSystem: indicators, thresholds, guardians, uncertainty, affected proposals
- Claim: claimant, about objects, evidence, counterclaims, review status, decision usage
- Issue: scope, claims, perspectives, proposals, attention state, readiness
- Proposal: affected objects, claims/evidence, scenarios, objections, decision method
- Decision: authority, process, rationale, unresolved objections, review date, supersession
- Commitment: parties, promise, constraints, fulfillment state, outcomes
- Flow: source, destination, quantity, timing, loss/waste, ecological impact
- Model: assumptions, inputs, uncertainty, scenarios, audit, decisions affected
- Policy: scope, rule text, authority, versions, effects, review obligations

## Civic Memory

Civic memory is the shared historical layer for Canopy. It is not a notification stream, audit afterthought, or module log.

### Civic Memory Requirements

The shell must provide:

- Object-local memory timelines
- Scope-level memory views
- Cross-scope memory search
- Decision packet history
- Supersession history
- Import/export/federation events
- Redacted event stubs
- Audit and integrity views
- Digests for human review

### Civic Memory Event Anatomy

Each memory item should expose:

- What changed
- Who or what acted
- When it occurred
- Which object was affected
- Which scope governed it
- Which authority, mandate, or policy allowed it
- Which claims, evidence, or decision packet supported it
- Which data stewardship rules shaped visibility
- Whether it supersedes or is superseded by another event
- Whether federation/export was involved

### Memory Filters

Core filters:

- Governance
- Claims/evidence
- Stewardship
- Allocation/accounting
- Ecology
- Care
- Flows
- Simulation/modeling
- Federation
- Integrity
- Learning

### Memory Rules

- Module-specific histories must be translated into Canopy civic memory events.
- Consequential mutations cannot bypass memory.
- Redaction must preserve the existence of accountable events when full detail is restricted.
- Memory should support forgiveness: distinguish institutional accountability from permanent personal stigmatization.

## Attention Inbox

The attention inbox is Canopy's shared queue for review, response, deliberation, stewardship, and learning. It is not a generic notification center.

### Attention Item Types

Attention items include:

- Threshold breached
- Claim challenged
- Evidence requested
- Review due
- Proposal ready for deliberation
- Decision blocked
- Guardian review required
- Commitment overdue
- Task blocked
- Appeal filed
- Conflict needs facilitation
- Model audit due
- Policy sunset approaching
- Federation conflict detected
- Data access request pending
- Care load warning
- Outcome review due

### Attention Item Anatomy

Each item shows:

- Attention type
- Affected object
- Scope
- Severity or urgency
- Due date or trigger
- Required role or authority
- Related claims/evidence
- Suggested next action
- Consequences of inaction
- Visibility and sensitivity state

### Attention Routing

Routing is based on:

- Role assignments
- Mandates
- Delegations
- Guardianship
- Stewardship responsibility
- Membership
- Consent and data access
- Object subscriptions
- Governance process participation

### Attention Rules

- Not every event becomes attention.
- Attention must be actionable, reviewable, dismissible, assignable, or deferrable.
- Attention generated by AI must identify its basis and uncertainty.
- Care-related attention must protect privacy and avoid coercive visibility.
- Attention should privilege ecological and governance triggers over engagement metrics.

## Capability Surfaces

Capability surfaces are task-oriented workspaces over the shared object graph.

### Capability Surface Anatomy

Each surface should include:

- Scope-aware entry state
- Relevant object lists, maps, or graphs
- Current attention for that capability
- Common commands
- Recent civic memory
- Governance hooks
- Claims/evidence hooks
- Data stewardship state
- Export/federation state where applicable

### Required Capability Surfaces

#### Reality Map

Shows places, commons, resources, living systems, flows, indicators, and spatial relationships. It supports observe and understand phases.

Must not become a surveillance map or treat administrative boundaries as the only real boundaries.

#### Commons Registry

Shows commons, resources, use rights, routines, tasks, stewardship assignments, policies, and obligations. It supports observe, coordinate, act, and learn phases.

#### Living Systems

Shows ecological objects, indicators, thresholds, guardians, evidence, uncertainty, restoration obligations, affected proposals, and review triggers.

#### Needs And Capabilities

Shows needs, capabilities, requests, offers, matches, commitments, constraints, and unresolved gaps. It coordinates without reducing needs to market demand.

#### Claims And Evidence

Shows claims, counterclaims, sources, evidence quality, review states, contested objects, and decision usage. It is a universal epistemic surface, not a separate "sensemaking app."

#### Deliberation

Shows issues, perspectives, proposals, objections, amendments, facilitation state, decision readiness, and affected scopes.

#### Agreements And Policies

Shows agreements, policies, versions, mandates, review dates, obligations, supersessions, and effects.

#### Allocation And Accounting

Shows allocations, budgets, commitments, ledger entries, use rights, reciprocity records, and disputes. It treats accounting as commitment memory, not capital accumulation.

#### Flows

Shows movement of food, water, energy, materials, money, care, labor, knowledge, waste, and attention across places, commons, projects, and living systems.

#### Simulation And Models

Shows models, assumptions, scenarios, uncertainty, sensitivity analysis, tradeoff surfaces, threshold effects, and model audits.

#### Care Coordination

Shows consented care tasks, care-load signals, anonymous holds, needs, commitments, and privacy-preserving coordination. It must preserve relational care from over-quantification.

#### Learning And Accountability

Shows outcomes, audits, retrospectives, indicator changes, policy effects, model performance, unresolved learning questions, and rule revision opportunities.

#### Federation

Shows export bundles, shared objects, sync state, reconciliation queues, federation agreements, forks, defederation processes, and cross-instance data stewardship.

## Command Palette

The command palette is the fast path for Canopy actions. It should feel like an operating-system command layer for civic work.

### Command Sources

Commands are generated from:

- Active scope
- Current object
- User role and mandates
- Data stewardship rules
- Object lifecycle state
- Capability manifests
- Open attention items
- Governance processes

### Command Categories

Core categories:

- Create object
- Add observation
- Add claim or evidence
- Challenge or request review
- Open issue
- Create proposal
- Make decision packet
- Create agreement or policy
- Create request or offer
- Create commitment
- Create allocation
- Assign task
- Record flow
- Record outcome
- Start retrospective
- Export or fork
- Request access

### Command Rules

- Commands must explain why they are available when authority is not obvious.
- Commands that affect rights, obligations, ecological state, claims, decisions, accounting, visibility, or federation must preview the civic memory event they will create.
- Commands should route users through the smallest sufficient workflow.
- Commands should use canonical verbs and object names rather than inherited product terminology.

## Map / Graph / List Triad

The triad is Canopy's shared way of navigating complex object sets.

### Map

Best for:

- Places
- Commons
- Resources
- Living systems
- Flows
- Infrastructure
- Spatial indicators
- Boundary disputes
- Ecological thresholds

Map rules:

- Show administrative, ecological, social, and federation boundaries as different layers.
- Sensitive locations must honor data stewardship agreements.
- Spatial objects must link back to object pages.

### Graph

Best for:

- Relationships
- Authority
- Claims/evidence
- Policy effects
- Flow dependencies
- Need-capability matching
- Mandates and delegations
- Federation links
- Causal maps and model assumptions

Graph rules:

- Relationship labels must use canonical verbs.
- Graph nodes must resolve to object pages.
- Graph views should allow local terms as labels while preserving canonical relationships.

### List

Best for:

- Review queues
- Object inventories
- Tasks
- Proposals
- Commitments
- Allocations
- Policies
- Memory events
- Search result sets

List rules:

- Lists must support scope, type, state, authority, attention, and visibility filters.
- Lists should be action-capable where permissions allow.
- Lists must not fork into module-specific table worlds.

### Triad Behavior

Users can switch between map, graph, and list views for the same object set without changing the underlying query.

Example:

- A watershed resilience scope can show threatened resources on a map.
- The same result set can show guardian, policy, and claim relationships as a graph.
- The same result set can show overdue reviews and commitments as a list.

## Decision Packets

Decision packets are the shell's structured way to make governance legible.

### Decision Packet Contents

A packet should include:

- Proposal or issue
- Affected objects
- Scope and authority
- Relevant claims
- Evidence and sources
- Counterclaims and unresolved disagreements
- Perspectives
- Ecological thresholds
- Guardian review
- Scenarios and model outputs
- Options considered
- Decision method
- Participation record
- Objections and amendments
- Final decision
- Agreements, policies, commitments, allocations, or obligations created
- Review date
- Outcome tracking plan

### Decision Packet Rules

- Decisions cite claims and evidence; they do not bury them in prose.
- AI summaries are reviewable artifacts, not authority.
- Packets must show what remains uncertain or contested.
- Packets must become civic memory and be linkable from every affected object.

Mobile is optimized for:

- Capture
- Review
- Attention response

## How Old Project Boundaries Disappear

The shell must actively prevent inherited project identities from becoming user-facing silos.

### Source-To-Shell Translation

| Historical source | Shell translation |
| --- | --- |
| CommonCredit offers, needs, transactions, disputes | Needs and Capabilities, Commitments, Allocation and Accounting, Appeals |
| ICOS spaces, issues, perspectives, delegations, civic memory | Scopes, Deliberation, Mandates, Decision Packets, Civic Memory, Federation |
| Sensemaking claims, sources, themes, reviews | Claims and Evidence across all object pages and capability surfaces |
| Stewardship resources, access rights, policies, maintenance, food flows | Commons Registry, Use Rights, Policies, Routines, Tasks, Flows |

### Product Boundary Removal Rules

Do not expose:

- CommonCredit app
- ICOS app
- Sensemaking app
- Stewardship app

Do expose:

- Objects
- Scopes
- Capabilities
- Civic memory
- Attention
- Decisions
- Claims and evidence
- Commitments and obligations
- Living-system context
- Federation and export

### Migration Labeling

Old project names may appear only in:

- Admin migration views
- Developer diagnostics
- Source lineage metadata
- Historical provenance inside civic memory
- Capability manifest documentation

They should not appear as primary navigation, user roles, page templates, or object categories.

## Data Stewardship In The Shell

Data stewardship is not a settings page. It must be visible wherever knowledge, privacy, consent, access, export, or federation matter.

### Shell-Level Data Signals

The shell should expose:

- Visibility level
- Sensitivity flags
- Consent state
- Access rules
- Exportability
- Federation state
- Retention rules
- Redaction summary
- Stewardship agreement link

### Data Stewardship Actions

Users may be able to:

- Request access
- Challenge visibility
- Change data stewardship agreement
- View redaction summary
- Export object
- Fork scope
- Revoke or revise consent where governance allows
- Appeal a data access decision

Rules:

- Transparency must not become surveillance.
- Privacy must not become unaccountable opacity.
- Sensitive redactions should preserve civic accountability through structured stubs where possible.

## Federation And Forkability In The Shell

Federation must be present from the beginning as IA, even if full sync capabilities mature later.

### Federation Surface

The shell should show:

- Which objects are local
- Which objects are federated
- Which scopes are shared
- Which data is exportable
- Which data is redacted, embargoed, or non-federating
- Sync status
- Reconciliation conflicts
- Federation agreements
- Fork and defederation paths

### Federation Rules

- Federation cannot erase local autonomy.
- Export must preserve provenance, access rules, schema version, and civic memory.
- Reconciliation conflicts should become attention items and, where consequential, governance issues.
- Defederation must have a visible, legitimate process.

## AI In The Shell

AI appears as assistance inside the shell, not as sovereign authority.

Appropriate AI uses:

- Summarizing object state
- Detecting unresolved claims
- Suggesting related evidence
- Extracting possible claims from sources
- Identifying attention patterns
- Matching needs and capabilities
- Surfacing relevant policies
- Comparing scenarios
- Drafting decision packet summaries

AI requirements:

- Show inputs used
- Show assumptions and uncertainty
- Distinguish AI summary from claim, evidence, model output, or decision
- Provide challenge, hide, revise, or request-review actions
- Respect data stewardship and scope permissions
- Write consequential AI-mediated changes to civic memory through human-confirmed workflows

## Integrity And Anti-Capture IA

The shell must help resist misinformation, factional capture, administrator overreach, data poisoning, and hidden scoring.

Required IA features:

- Claim challenge paths
- Evidence provenance
- Bot/spam review queues
- Model audit surfaces
- Moderation memory
- Data stewardship appeals
- Governance of platform defaults
- Federation conflict visibility
- Export and fork rights
- No hidden person scoring or quiet eligibility profiles

## Example End-To-End Flow

Scenario: a watershed guardian flags that a stream indicator has crossed a governance-trigger threshold.

1. Indicator update writes to civic memory.
2. Threshold breach creates an attention item for guardians, stewards, and relevant facilitators.
3. The affected LivingSystem object page shows current state, claim/evidence context, threshold class, and affected policies.
4. Search reveals related resources, flows, projects, and prior decisions in the same watershed scope.
5. The graph view shows relationships among the stream, upstream resources, policies, claims, and open commitments.
6. A guardian opens an issue from the object page.
7. Deliberation surface gathers perspectives, evidence, scenarios, and possible proposals.
8. A decision packet cites claims, evidence, model uncertainty, ecological thresholds, and affected commitments.
9. A decision creates an agreement, maintenance obligations, and allocation.
10. Tasks and flows are coordinated through capability surfaces.
11. Outcomes update indicators and trigger a retrospective.
12. The whole sequence remains visible through civic memory across all affected objects.

No user needed to know whether this involved inherited Stewardship, ICOS, Sensemaking, or CommonCredit concepts. They moved through Canopy objects, scopes, attention, capability surfaces, governance, and memory.

## Canopy Shell Compliance Checklist

A shell experience is Canopy-compliant if:

- It uses one sign-in and one identity/authority model.
- It exposes one global scope switcher.
- It exposes one global object search.
- It uses the universal object page pattern.
- It writes consequential events to civic memory.
- It distinguishes claims, evidence, model outputs, AI summaries, and decisions.
- It makes governance and authority visible before consequential action.
- It supports the attention inbox as a shared review/action queue.
- It exposes capability surfaces as lenses over the shared object graph.
- It supports map, graph, and list views over the same object sets.
- It makes living systems first-class where applicable.
- It makes data stewardship visible at object, scope, search, and federation levels.
- It supports export, fork, reconciliation, and defederation concepts.
- It keeps old project names out of primary user navigation.
- It helps users complete the cybernetic loop from observation to learning.

## Non-Negotiables

- No separate module histories.
- No separate module identities.
- No old product names as top-level navigation.
- No consequential action without authority context.
- No decision without traceable evidence and memory.
- No material coordination that hides ecological context.
- No AI recommendation that cannot be inspected or challenged.
- No search result that obscures claim/evidence/decision distinctions.
- No interface pattern that treats care, people, or living systems as mere inventory.

## Summary

The Canopy unified shell is the IA layer that makes the reference architecture inhabitable. It turns the kernel, object graph, claims/evidence layer, governance spine, civic memory, and federation protocols into one coherent operating environment.

If the shell works, users experience Canopy as a civic and ecological commons infrastructure: scoped, searchable, governable, contestable, memorable, federated, and alive to feedback.

If the shell fails, the ecosystem will regress into separate apps with shared branding.

The shell's job is to make that regression impossible.
